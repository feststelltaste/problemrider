## How to Apply ◆

> Legacy databases were designed for original data volumes and access patterns that often differ dramatically from current production load — systematic query optimization recovers performance without requiring expensive hardware upgrades or risky schema rewrites.

- Enable slow query logging as the first step (`log_min_duration_statement` in PostgreSQL, `slow_query_log` in MySQL) and let it run for at least one full business cycle to capture the real query workload, not just what developers assume is slow.
- Prioritize optimization by impact: a query that runs 10,000 times per day and takes 300ms is more important than one that runs once and takes 10 seconds — calculate total time consumed per query, not just worst-case duration.
- Use EXPLAIN (ANALYZE) on every candidate query before making changes; legacy databases frequently have stale table statistics that cause the query planner to choose a sequential scan on a multi-million-row table when an index scan would take milliseconds — running ANALYZE first can immediately fix poor query plans without any schema changes.
- Add indexes to match the actual WHERE, JOIN, and ORDER BY patterns of the most expensive queries; in legacy systems, many tables were created with only a primary key index and rely on full table scans for every other access pattern.
- Identify and rewrite N+1 query patterns generated by ORMs or handwritten loops in application code; these are among the most common performance killers in legacy Java EE and older Rails applications where lazy loading was the unexamined default.
- Replace `SELECT *` with named column lists throughout the codebase; legacy systems often fetch entire wide rows when only two or three columns are needed, wasting I/O, memory, and network bandwidth.
- Tune database server memory settings to reflect the actual hardware available; legacy databases frequently run with default buffer pool sizes (128MB) on servers with 64GB of RAM, leaving most of the machine's memory idle while the database hammers disk.
- Separate read-heavy reporting queries onto read replicas so they do not compete with transactional queries for the same database resources — a change that legacy architectures almost never include but that dramatically improves both workloads.

## Tradeoffs ⇄

> Database query optimization in legacy systems delivers some of the largest performance gains available at the lowest structural risk, but it requires ongoing attention as data volumes and query patterns continue to evolve.

**Benefits:**

- Well-placed indexes can reduce individual query times from seconds to milliseconds — improvements of 100x or more — without touching application logic or risking data integrity.
- Optimization defers the need for expensive database migrations, sharding, or hardware upgrades by making existing infrastructure far more efficient, extending the productive life of legacy systems.
- Slow query analysis reveals which parts of the application generate the most database load, directing refactoring effort to where it has the highest payoff rather than where developers intuit problems.
- Fixing N+1 patterns and replacing SELECT * often requires changes only at the data access layer, leaving business logic untouched — low-risk improvements in codebases where touching business logic is considered dangerous.
- Reduced lock contention from faster queries improves overall system throughput under concurrent load, addressing the "works fine with 20 users, breaks with 200 users" complaint common in aging web applications.

**Costs and Risks:**

- Indexes slow write operations and consume disk space; over-indexing legacy tables with high insert rates — common in logging or event-sourcing patterns — can make writes dramatically slower.
- Query optimization for current access patterns may not suit future patterns; a covering index built for today's most common query becomes dead weight if the application changes how it retrieves data.
- Configuration tuning (buffer pools, work memory, checkpoint intervals) requires understanding of database internals that is rarely present in teams that inherited a legacy system without its original architects.
- Partitioning large legacy tables to improve query performance introduces cross-partition query complexity and requires careful coordination with any foreign key relationships and application queries that span the partition boundary.
- Optimization effort can become a substitute for addressing the underlying architectural problems (e.g., shared databases, missing caching layers) that cause excessive database load in the first place.

## Examples

> The following scenarios illustrate query optimization applied in realistic legacy system contexts.

A mid-sized e-commerce company had a legacy PHP application backed by a MySQL database that had grown from 100,000 to 40 million product records over a decade. The product search page had degraded from a 200ms response to an 8-second timeout under normal load. Enabling the slow query log revealed that the search query performed a full table scan on the products table on every request because the WHERE clause combined columns in an order that did not match any existing index. Adding a composite index on `(category_id, status, price)` to match the query's filter pattern reduced the search query to 18ms without changing a single line of application code.

A government agency ran nightly batch jobs against an Oracle database that had been growing for twelve years and now held 800 million rows in its primary transaction table. The jobs, written in the early 2000s, used `SELECT *` to fetch full rows before filtering in application code, and had been taking 14 hours to complete — overrunning into business hours. A review of the execution plans showed that rewriting three queries to select only the needed columns and adding two partial indexes on the subset of active records reduced the batch runtime to under 3 hours, meeting the overnight window without any changes to the batch logic structure.

An insurance company's legacy claims management system experienced periodic slowdowns that correlated with high concurrent user counts. Investigation with EXPLAIN ANALYZE revealed that a critical query joining four tables had an estimated row count of 200 when the actual count during peak periods was 2 million, causing the planner to choose a nested-loop join strategy that worked at small scale but collapsed under load. Running ANALYZE to refresh statistics and adjusting `default_statistics_target` for the skewed columns caused the planner to switch to a hash join, cutting the query time from 45 seconds to 400ms and eliminating the periodic slowdowns entirely.
