## How to Apply ◆

> Legacy systems often lack mechanisms to verify the authenticity and integrity of code, documents, and data — making it impossible to detect tampering or prove authorship. Digital signatures provide cryptographic proof that content has not been modified since it was signed by a known party.

- Implement code signing for all deployable artifacts (binaries, packages, container images, scripts) so that the deployment pipeline can verify that only authorized, unmodified code reaches production.
- Sign database migrations and configuration changes so that applied changes can be verified against their signed originals, detecting unauthorized modifications.
- Use digital signatures for inter-system data exchange to ensure that messages received from external partners have not been tampered with in transit and originate from the claimed sender.
- Implement document signing for audit-critical documents, contracts, and reports generated by the legacy system to provide non-repudiation and tamper evidence.
- Establish a key management process for signing keys: generate keys using hardware security modules, distribute public keys through trusted channels, implement key rotation schedules, and define procedures for key compromise response.
- Verify signatures at the point of consumption, not just at the point of creation. Every component that receives signed content should independently verify the signature before processing it.

## Tradeoffs ⇄

> Digital signatures provide tamper detection and authorship proof, but they require PKI infrastructure and key management discipline.

**Benefits:**

- Detects unauthorized modification of code, data, and documents by providing cryptographic integrity verification.
- Provides non-repudiation — the signer cannot deny having signed the content, which is important for legal and compliance purposes.
- Prevents supply chain attacks by ensuring that deployed code is the same code that was built and approved.
- Enables automated trust decisions where systems can verify the authenticity of received data without human intervention.

**Costs and Risks:**

- Key management adds operational complexity — compromised signing keys can be used to sign malicious content, and lost keys prevent verification of previously signed content.
- Signature verification adds processing time to every operation that involves signed content.
- Transitioning legacy systems to signed artifacts requires tooling changes in build pipelines, deployment processes, and data exchange interfaces.
- Digital signatures prove integrity and authenticity but not correctness — signed malware is still malware.

## Examples

> The following scenarios illustrate how digital signatures protect legacy system integrity.

A legacy deployment process copies compiled Java WAR files from a shared network drive to production application servers. An attacker who gains access to the network drive replaces a legitimate WAR file with a modified version containing a backdoor. The modification goes undetected because there is no integrity verification in the deployment process. The team implements code signing: the CI/CD pipeline signs each build artifact with a GPG key stored in a hardware security module, and the deployment script verifies the signature before deploying. When the attacker again modifies a WAR file on the network drive, the deployment script detects the invalid signature and refuses to deploy, alerting the security team to the compromise attempt.

A legacy EDI (Electronic Data Interchange) system exchanges purchase orders and invoices with suppliers via unprotected file transfers. A man-in-the-middle attack modifies invoice amounts during transit, redirecting payments to the attacker's bank account. The team implements digital signatures for all EDI documents: the sender signs each document with their private key, and the receiver verifies the signature before processing. When a subsequent modification attempt occurs, the signature verification fails, and the modified invoice is rejected. The sender is notified to retransmit, and the security team investigates the network path to identify the interception point.
